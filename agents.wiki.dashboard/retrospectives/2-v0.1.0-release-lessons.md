# Release Lessons: dashboard parity mobile project scope and G9 recovery

## What went well
- Scope and contracts were completed across project selector, token pagination metadata, and Quality surfaces after a continuation and targeted fixes.
- Security findings were closed before final approvals, removing a critical supply-chain dependency risk.
- G9 preview failures were corrected with a minimal CSS cascade fix, then browser evidence captured.
- Designer and code-review checks improved final behavior against mobile layout and parity expectations.

## What went wrong
- UI-heavy planning initially skipped design gate and used mixed profile metadata, adding a late planning correction cycle.
- Browser-quality smoke tooling was inconsistent, causing temporary QA blocks and status drift despite evidence arriving later.
- Initial G9 mobile fix passed style-regex checks but failed visual layout due cascade order.
- Route and API contract gaps (project-scoped live + monthly token flow) surfaced only after implementation, increasing rework loops.

## Key lessons
- Treat security-triggered escalations as mandatory full retro triggers and make them explicit in handoff packets.
- For preview-critical layout gates, require viewport-accurate evidence, not text/regression checks alone.
- Keep shared project-query and token-pagination contracts tested at both API and route layers before final signoff.
- Keep temporary QA tooling dependency assumptions explicit and reproducible to prevent environment false-blocks.
