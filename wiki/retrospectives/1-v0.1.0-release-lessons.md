# Release Lessons: provider-limits native + `/limits` data fidelity

## What went well
- PIDEX-native provider-limit flow moved to local state and UI/API now shows `codex` and `codex-spark` rows when source data exists.
- Recommendation fields/UX were removed from probe, API, and profile routes, matching objective.
- Security controls improved after review: dashboard defaults now loopback, public-bind path requires token, and auth tests now cover spoofed-host/public-bind/host-origins.
- Orchestrator-level browser smoke confirmed desktop and mobile `/limits` rendering with seeded native records.

## What went wrong
- Early implementation had a security-control gap (host-based locality trust) and required extra review/remediation cycles.
- Security and dependency checks surfaced a critical advisory; package pin and lockfile refresh were needed late.
- QA chain had status drift: final QA outcome evidence and gate result were inconsistent in one intermediary file.

## Key lessons
- Treat network-facing route trust decisions as explicit modes: local-only default versus explicit public-bind with token.
- Require fixture-backed and browser-visible checks for plan-critical provider rows before passing code review.
- Keep status metadata synchronized with evidence at each handoff to prevent gate ambiguity.
- Separate temporary QA tooling paths from runtime dependency manifests to reduce recurring dependency-noise.
